## 2026-05-10 — Per-step cancel-cleanup hooks in recipes

- **Idea:** Add an **optional, per-step** cancel-cleanup field on `.invokeCommand` and `.invokeRecipe` steps in the recipe DSL, alongside the existing recipe-level `FleetRecipeDescriptor.cancelRecipe`. Authors should be able to declare step-local cleanup, recipe-level cleanup, both, or neither — it must remain optional in every shape.
- **Context:** `Sources/GuardianHQ/Systems/Fleet/Subsystems/RecipesCatalogue/` — `FleetRecipeStep.swift` (new optional field per case), `FleetRecipeBodyParser.swift` (validate the new field + reject composite or cleanup-of-cleanup targets), `FleetRecipeRunner.swift` (per-step cleanup runs before recipe-level cleanup when both are declared). README "Layer 1 runner" section documents the current single-cleanup behaviour and is the right place to extend.
- **Notes:** Resolution rule when both declared: per-step cleanup first, then recipe-level. Parser must apply the same atomic-only / no-self-reference constraints used today for `cancelRecipe`. No DSL or runtime work in v1 — v1 ships only the recipe-level field.

## 2026-05-11 — Mission map points: live RoE / replan when coordinates or closed state change (MCR / MRE)

- **Idea (former `MissionPointsTodo.md` §11):** When vehicles are **actively navigating** toward a **runtime** map point, **coordinate moves** (operator drag on the live map) should trigger MRE **detection**, optional **reroute / replan**, and **operator confirmation** under Rules of Engagement before or after applying the move. When a point is **closed** or **reopened** while it is an active rally/extraction target, MRE should evaluate in-flight use, **reroute or hold** per policy, and surface **operator prompts** under RoE. Today only **logging + state persistence** run (`MissionRunEnvironment` apply/update/close APIs); there is no loop that ties live point mutations to in-flight fleet behaviour.
- **Context:** `MissionRunEnvironment.applyRuntimeMissionPointUpdate`, `applyRuntimeMissionPointSetClosed`, MC-R map points overlay. Fleet work is driven through **`MissionRunExecutionSubsystem`** and **`MissionRunCommandSubsystem`** — today the path is effectively **fire-and-forget** from the executor’s perspective: commands and recipe invocations are **dispatched** without the Mission Control core **holding a durable, per-flight subscription** to recipe **step progress** so it can automatically inject **mid-run replans** when the world model (points, vehicles, RoE) changes. That architectural gap means the §11 behaviours are **not implementable as polish on top of the current stack**; they need an explicit **codebase change** (executor / planner / observer boundary).
- **Notes:** Desirable end states include either **(a)** upgrading MRE execution so something in the run stack **observes** catalogue / `FleetRecipeRunner` progress and can **branch or cancel-and-replan** when points move or close, or **(b)** leaning on **Paladin** (or a similar assistant) to **subscribe** to recipe / telemetry signals and **propose mutations** (plan deltas, operator confirms) when replanning is required — without duplicating fleet MAVLink paths. Until one of those exists, document operator limits (e.g. “do not move a point vehicles are using without manual intervention”) in product copy if needed.
